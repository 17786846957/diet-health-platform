# Full Review — 一键审查+修复+验证

按固定流程执行完整审查，确保代码库处于可交付状态。

## 执行流程（严格按顺序，不可跳步）

### Phase 1: 代码质量审查
1. 后端：扫描安全漏洞（硬编码密钥、SQL 注入、权限绕过、NPE 风险）
2. 前端：扫描 Bug（响应数据处理、内存泄漏、生命周期问题）
3. 记录所有发现，按严重度排序：Critical > High > Medium > Low

### Phase 2: 修复
1. 先修复 Critical 和 High 级别问题
2. 再修复 Medium 级别问题
3. Low 级别问题记录但不自动修复（除非用户要求）
4. 每个修复用 Edit 工具精确修改，不重写整个文件

### Phase 3: 编译验证
1. 后端：`cd diet-health-backend && mvn compile -q`
2. 前端：`cd diet-health-frontend && npm run build`
3. 任一失败 → 立即修复编译错误，不继续下一步

### Phase 4: 测试验证
1. 后端：`cd diet-health-backend && mvn test`
2. 任一测试失败 → 读失败测试和对应源文件 → 修复源代码 → 重跑
3. 最多 3 轮修复循环

### Phase 5: 输出报告
格式：
```
审查报告
========
发现问题: X 个
已修复:   X 个
编译状态: PASS/FAIL
测试状态: X/Y 通过

修复清单:
1. [严重度] 文件:行号 — 描述
2. ...
```

## 注意事项
- 不改测试文件，除非测试本身有断言 Bug
- 修复时遵循 Karpathy 原则：最小改动、不加未要求的功能
- 每次 Edit 后不单独编译，在 Phase 3 统一验证
