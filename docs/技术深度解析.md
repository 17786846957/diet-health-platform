## 一、系统架构决策 — 前后端分离 B/S 架构

### 1.1 技术选型

| 属性 | 内容 |
|------|------|
| **技术** | 前后端分离架构 (Frontend-Backend Separation) |
| **分类** | 架构模式 |
| **角色** | 系统整体架构，决定前后端如何协作 |

### 1.2 为什么选择

**项目约束：**
- 毕业设计，一个人同时写前端和后端，需要能独立开发和调试
- 需要数据可视化（ECharts 图表），纯后端渲染不现实
- 答辩时需要展示完整的 UI 交互效果

**匹配原因：**
- 前后端分离后，前端用 Vite 热更新，改 UI 不用重启后端
- 后端只返回 JSON，不关心页面长什么样，职责清晰
- RESTful API 是标准协议，答辩时容易解释

**权衡取舍：**
```
✅ 选择了：前后端分离，独立开发部署，RESTful API 通信
❌ 放弃了：服务端渲染（SSR）的首屏加载速度和 SEO 优势
⚖️ 这个交换在本项目中是值得的，因为：内部管理系统不需要 SEO，
   首屏加载速度在内网环境下不是瓶颈
```

### 1.3 核心原理

前后端分离的本质是**关注点分离**（Separation of Concerns）。传统 MVC 模式下，后端既管业务逻辑又管页面渲染，前后端代码耦合在一起。分离后：

```
传统 MVC:
  浏览器 → 请求 → 后端(Controller + 模板渲染) → 完整 HTML → 浏览器

前后端分离:
  浏览器 → 请求 → 后端(只返回 JSON) → 浏览器 → 前端(JS 渲染页面)
```

关键变化：**渲染权从服务器转移到了浏览器**。前端是一个独立的 SPA（Single Page Application），通过 AJAX 请求后端 API 获取数据，自己负责渲染页面。

### 1.4 关键概念

**RESTful API 契约**

前后端之间的唯一约定是 API 接口。项目用统一的 `R` 对象封装响应：

```java
// 后端统一响应格式
public class R<T> {
    private int code;       // 状态码：200 成功，其他失败
    private String message; // 提示信息
    private T data;         // 业务数据
}
```

前端拿到这个 JSON 后，统一判断 `code === 200` 再处理 `data`。这个约定让前后端可以**并行开发**——前端先用 Mock 数据开发页面，后端写好接口后直接对接。

**axios 拦截器 — 前后端的"桥梁"**

项目在 `request.js` 中用 axios 拦截器统一处理认证和错误：

```javascript
// 响应拦截器 — 统一处理 401 和业务错误
instance.interceptors.response.use(
  (response) => {
    const { code, message, data } = response.data
    if (code !== 200) {
      ElMessage.error(message || '请求失败')
      return Promise.reject(new Error(message))
    }
    return data  // 直接返回 data，调用方不用再解包
  },
  (error) => {
    if (error.response?.status === 401) {
      // Token 过期，跳转登录页
      useUserStore().logout()
      router.push('/login')
    }
    return Promise.reject(error)
  }
)
```

这个拦截器做了三件事：
1. 统一解包 `R` 对象，调用方直接拿到 `data`
2. 业务错误统一弹窗提示
3. 401 自动跳转登录页

### 1.5 常见陷阱

| # | 陷阱 | 后果 | 避免/解决方法 |
|---|------|------|--------------|
| 1 | 前后端接口不同步 | 前端拿到的数据结构和预期不一致 | 用统一的 `R` 对象 + DTO 约束 |
| 2 | 前端直接拼 SQL 字符串 | XSS 注入风险 | 前端只传参数，后端用 LambdaQueryWrapper |
| 3 | 忘记 CORS 配置 | 浏览器跨域请求被拦截 | CorsConfig 允许前端域名 |

### 1.6 替代方案对比

| 对比维度 | ✅ **前后端分离** | 🔶 **服务端渲染 (SSR)** | 🔶 **Thymeleaf 模板** |
|---------|------------------|------------------------|----------------------|
| **核心技术** | Vue 3 SPA + REST API | Next.js / Nuxt.js | Spring MVC + 模板引擎 |
| **核心优势** | 独立开发，交互流畅 | 首屏快，SEO 好 | 简单，不需要 Node.js |
| **主要劣势** | 首屏需加载 JS | 前后端仍耦合 | 页面刷新，交互差 |
| **适用场景** | 内部管理系统 | 面向公众的网站 | 简单 CRUD 后台 |
| **学习曲线** | 中等 | 陡峭 | 平缓 |
| **本项目适配** | ✅ 最优 | 过度 | 无法满足图表需求 |

> **对比总结：** 内部管理系统不需要 SEO，前后端分离的开发效率和交互体验是最优选择。

### 1.7 优越性

1. **开发效率高** — 前端 Vite 热更新 < 100ms，改代码即时看到效果；后端 Spring Boot DevTools 自动重启
2. **职责清晰** — 后端只管业务逻辑和数据，不关心页面；前端只管 UI 和交互，不关心数据库
3. **部署灵活** — 前端静态文件可以放 CDN，后端可以水平扩展，Docker Compose 一条命令部署

### 1.8 知识延伸

**思想迁移：** 关注点分离的思想无处不在——MVC 分离模型和视图、微服务分离业务域、CQRS 分离读写。核心都是：**每个组件只做一件事，通过明确的接口协作**。

## 二、安全架构 — JWT httpOnly Cookie 认证方案

### 2.1 技术选型

| 属性 | 内容 |
|------|------|
| **技术** | JWT (JSON Web Token) + httpOnly Cookie |
| **分类** | 认证协议 / 安全架构 |
| **角色** | 用户身份认证和 Token 安全传输 |

### 2.2 为什么选择

**项目约束：**
- 前后端分离，无法使用传统的 Session 认证（Session 绑定服务器内存）
- 需要防 XSS 攻击（攻击者注入脚本窃取 Token）
- 需要防 CSRF 攻击（攻击者伪造请求）

**权衡取舍：**
```
✅ 选择了：JWT httpOnly Cookie，兼顾无状态和安全
❌ 放弃了：localStorage 存 Token（更简单但不防 XSS）
⚖️ 安全是饮食健康系统的基本要求，用户的健康数据不应被窃取
```

### 2.3 核心原理

JWT 是一个自包含的 Token，结构为 `Header.Payload.Signature`：

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInVzZXJuYW1lIjoiYWRtaW4ifQ.xxx

Header:    {"alg": "HS256"}              — 签名算法
Payload:   {"userId":1, "username":"admin"} — 用户信息（可解码但不可篡改）
Signature: HMAC-SHA256(header+"."+payload, secret) — 签名验证
```

认证流程：

```
1. 用户登录 → 后端验证密码 → 生成 JWT → 写入 httpOnly Cookie
2. 后续请求 → 浏览器自动携带 Cookie → 后端解析 JWT → 获取 userId
3. Token 剩余 < 2 小时 → 自动续期 → 返回新 Token
```

### 2.4 关键概念

**httpOnly Cookie vs localStorage**

```
localStorage:
  ✅ JavaScript 可读写
  ❌ XSS 攻击可通过 document.cookie 或 localStorage.getItem() 窃取
  ❌ 需要手动在每个请求中添加 Authorization Header

httpOnly Cookie:
  ✅ JavaScript 不可读（document.cookie 拿不到 httpOnly 的值）
  ✅ 浏览器自动在每个请求中携带
  ❌ 需要防 CSRF（SameSite 属性解决）
```

项目选择了 httpOnly Cookie，这是**安全优先**的决策。

**Token 自动续期**

```java
// JwtAuthenticationFilter 中的续期逻辑
if (jwtUtil.isTokenExpiringSoon(token)) {  // 剩余 < 2 小时
    String newToken = jwtUtil.generateToken(userId, username, role);
    JwtUtil.addTokenCookie(request, response, newToken);  // 写入新 Cookie
}
```

活跃用户永远不需要重新登录，但不活跃用户 24 小时后自动过期。

### 2.5 常见陷阱

| # | 陷阱 | 后果 | 避免/解决方法 |
|---|------|------|--------------|
| 1 | Token 存 localStorage | XSS 攻击可窃取 Token | 用 httpOnly Cookie |
| 2 | 不设 SameSite 属性 | CSRF 攻击可伪造请求 | SameSite=Lax |
| 3 | JWT Secret 硬编码 | 代码泄露后攻击者可伪造 Token | 环境变量注入，开发/生产分离 |

### 2.6 替代方案对比

| 对比维度 | ✅ **JWT httpOnly Cookie** | 🔶 **Session** | 🔶 **OAuth 2.0** |
|---------|--------------------------|---------------|-----------------|
| **核心优势** | 无状态 + 安全 | 服务端可控 | 第三方登录 |
| **主要劣势** | 无法主动失效 | 需要共享存储 | 复杂度高 |
| **适用场景** | 前后端分离单体 | 传统 Web 应用 | 需要第三方登录 |
| **学习曲线** | 中等 | 平缓 | 陡峭 |

> **对比总结：** 前后端分离的单体应用，JWT httpOnly Cookie 是最佳平衡点。

## 三、数据隔离 — member_id 字段模式

### 3.1 技术选型

| 属性 | 内容 |
|------|------|
| **技术** | 关键字段数据隔离 (Column-based Multi-tenancy) |
| **分类** | 数据库设计模式 |
| **角色** | 同一账号下多成员数据隔离 |

### 3.2 为什么选择

**项目约束：**
- 一个账号需要管理多个家庭成员的饮食数据
- 查询时只返回当前成员的数据
- 不能让不同成员的数据互相干扰

**权衡取舍：**
```
✅ 选择了：单表 + member_id 字段
❌ 放弃了：每个成员一张独立表（拆表方案）
⚖️ 拆表会导致查询复杂（UNION ALL），而且新增成员需要建表，运维成本高
```

### 3.3 核心原理

数据隔离有三种常见方案：

```
方案 1：独立数据库 — 每个租户一个数据库（SaaS 级隔离）
方案 2：独立表 — 每个成员一张表（表级隔离）
方案 3：共享表 + 隔离字段 — 所有成员数据在同一张表，用字段区分（字段级隔离）
```

项目选了方案 3，这是**最轻量**的方案。核心代码：

```java
// 统一的条件构建方法 — 所有查询都用这个
private LambdaQueryWrapper<DietRecord> userMemberWrapper(
        Long userId, Long memberId) {
    LambdaQueryWrapper<DietRecord> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(DietRecord::getUserId, userId);  // 先按用户过滤
    if (memberId == null) {
        wrapper.isNull(DietRecord::getMemberId);  // null = 主用户
    } else {
        wrapper.eq(DietRecord::getMemberId, memberId);  // 有值 = 家庭成员
    }
    return wrapper;
}
```

这个方法封装了隔离逻辑，调用方不需要关心"主用户"和"家庭成员"的区别。

### 3.4 关键概念

**NULL 的特殊含义**

`member_id = NULL` 不是"没有数据"，而是"主用户的数据"。这是一个**约定**，不是数据库的 NULL 语义。好处是：
- 主用户不需要在 family_member 表中创建记录
- 查询逻辑统一：`memberId == null ? isNull : eq`

**前端的成员切换**

```javascript
// Pinia store
const activeMember = ref(null)  // null = 主用户
const activeMemberId = computed(() => activeMember.value?.id || null)
const activeProfile = computed(() => activeMember.value || user.value)

// 切换成员后，router-view 用 key 强制重新挂载
<router-view :key="route.path + activeMemberId" />
```

切换成员时，`activeMember` 变化 → `activeMemberId` 变化 → `router-view` 的 key 变化 → 页面重新挂载 → 重新请求数据。

### 3.5 常见陷阱

| # | 陷阱 | 后果 | 避免/解决方法 |
|---|------|------|--------------|
| 1 | 忘记在查询条件中加 memberId | 返回所有成员的数据，数据泄露 | 所有查询统一用 userMemberWrapper |
| 2 | 删除成员时不级联删除记录 | 孤儿记录，查询出错 | 外键 ON DELETE CASCADE |
| 3 | 前端切换成员后不刷新页面 | 显示上一个成员的数据 | router-view 动态 key 强制重挂载 |

### 3.6 替代方案对比

| 对比维度 | ✅ **member_id 字段** | 🔶 **独立表** | 🔶 **独立数据库** |
|---------|---------------------|--------------|------------------|
| **复杂度** | 低 | 中 | 高 |
| **查询效率** | 高（索引优化） | 中（UNION ALL） | 高（单库小） |
| **新增成员** | INSERT 一行 | CREATE TABLE | CREATE DATABASE |
| **数据隔离** | 依赖代码逻辑 | 物理隔离 | 完全隔离 |
| **适用场景** | 成员数 < 100 | 成员数 100-10000 | SaaS 多租户 |

> **对比总结：** 家庭场景下成员数 < 10，字段级隔离是最简方案。

## 四、推荐算法 — 六维度加权评分

### 4.1 技术选型

| 属性 | 内容 |
|------|------|
| **技术** | 多维度加权评分算法 (Weighted Scoring Model) |
| **分类** | 推荐系统 / 算法设计 |
| **角色** | 根据用户状态推荐适合的食物 |

### 4.2 为什么选择

**项目约束：**
- 毕业设计，算法需要可解释（答辩时评委能听懂）
- 数据量小（230+ 种食物），不需要复杂的协同过滤
- 需要考虑多个因素，不能只看热量

**权衡取舍：**
```
✅ 选择了：加权评分，简单可解释
❌ 放弃了：协同过滤（需要大量用户行为数据）
⚖️ 单用户系统没有"其他用户"的数据，协同过滤无法使用
```

### 4.3 核心原理

加权评分的本质是**多准则决策**（Multi-Criteria Decision Making）。把多个评价维度归一化到 [0, 1] 区间，乘以权重后求和：

```
总分 = Σ(维度分数 × 权重)

本项目：
  总分 = 营养适配度 × 0.30
       + 餐次适配度 × 0.25
       + 食物健康度 × 0.20
       + 历史偏好   × 0.15
       + 收藏偏好   × 0.05
       + 分类多样性 × 0.05
```

### 4.4 关键概念

**营养适配度 (30%) — 最核心的维度**

```java
// 计算食物对当前营养缺口的匹配程度
double calculateNutritionScore(Food food, double[] gaps) {
    // gaps[0]=热量缺口, gaps[1]=蛋白质缺口, gaps[2]=脂肪缺口, gaps[3]=碳水缺口
    double proteinMatch = Math.min(food.getProtein() / 20.0, 1.0);  // 归一化
    double fatMatch = Math.min(food.getFat() / 15.0, 1.0);
    double carbsMatch = Math.min(food.getCarbs() / 50.0, 1.0);

    // 哪个缺口最大，就给哪个更高的权重
    double maxGap = Math.max(gaps[1], Math.max(gaps[2], gaps[3]));
    if (maxGap == gaps[1]) return proteinMatch * 0.5 + fatMatch * 0.2 + carbsMatch * 0.3;
    if (maxGap == gaps[2]) return proteinMatch * 0.2 + fatMatch * 0.5 + carbsMatch * 0.3;
    return proteinMatch * 0.3 + fatMatch * 0.2 + carbsMatch * 0.5;
}
```

逻辑：先找出用户哪个营养素缺口最大，然后优先推荐能填补该缺口的食物。

**餐次适配度 (25%) — 不同餐次适合不同食物**

```java
// 早餐适合主食/蛋奶，午晚餐适合肉类/蔬菜
double calculateMealFitScore(Food food, String mealType) {
    String category = food.getCategory();
    if ("breakfast".equals(mealType)) {
        if ("主食".equals(category) || "蛋奶".equals(category)) return 1.0;
        if ("饮品".equals(category)) return 0.8;
        return 0.3;  // 早餐推荐肉类不太合适
    }
    // lunch/dinner: 肉类和蔬菜得高分
    if ("肉类".equals(category) || "蔬菜".equals(category)) return 1.0;
    return 0.5;
}
```

**食物健康度 (20%) — 高蛋白低热量加分**

```java
double calculateHealthScore(Food food) {
    double proteinScore = Math.min(food.getProtein() / 20.0, 1.0) * 0.5;
    double fiberScore = Math.min(food.getFiber() / 5.0, 1.0) * 0.3;
    double junkPenalty = 0;
    if (food.getCalories() > 400 && food.getProtein() < 5) {
        junkPenalty = 0.3;  // 高热量低蛋白 = 垃圾食品，惩罚
    }
    return Math.max(0, Math.min(1.0, proteinScore + fiberScore + 0.2 - junkPenalty));
}
```

### 4.5 常见陷阱

| # | 陷阱 | 后果 | 避免/解决方法 |
|---|------|------|--------------|
| 1 | 权重加起来不等于 1.0 | 评分范围不可控 | 每次修改权重后检查总和 |
| 2 | 维度分数没有归一化 | 某个维度主导结果 | 每个分数都映射到 [0, 1] |
| 3 | 只考虑营养不考虑口味 | 推荐的食物用户不爱吃 | 加入历史偏好和收藏维度 |

### 4.6 替代方案对比

| 对比维度 | ✅ **加权评分** | 🔶 **协同过滤** | 🔶 **深度学习** |
|---------|---------------|----------------|----------------|
| **核心优势** | 简单可解释 | 利用群体智慧 | 自动特征学习 |
| **主要劣势** | 权重需手动调 | 需要大量用户数据 | 不可解释，需要 GPU |
| **适用场景** | 单用户/小系统 | 大平台（百万用户） | 大平台 + 复杂场景 |
| **数据需求** | 低（单用户历史） | 高（用户-物品矩阵） | 极高 |
| **本项目适配** | ✅ 最优 | 无法使用 | 过度 |

> **对比总结：** 单用户系统没有协同过滤的数据基础，加权评分是最务实的选择。

## 五、数据库设计 — 11 张表 + MyBatis-Plus ORM

### 5.1 技术选型

| 属性 | 内容 |
|------|------|
| **技术** | MySQL 8.0 + MyBatis-Plus 3.5.5 |
| **分类** | 关系型数据库 + ORM 框架 |
| **角色** | 数据持久化和查询 |

### 5.2 为什么选择

**项目约束：**
- 毕业设计，数据库需要成熟稳定
- 表结构简单（11 张表），不需要复杂的分布式特性
- 需要快速开发 CRUD，不想写大量 SQL

### 5.3 核心原理 — 表关系

```
user (1) ──→ (N) diet_record (1) ──→ (N) diet_record_detail (N) ←── (1) food
  │                    │
  │                    └──→ (0..1) family_member
  │
  ├──→ (N) food_favorite (N) ←── (1) food
  ├──→ (N) water_record
  ├──→ (N) exercise_record
  ├──→ (N) weight_record
  ├──→ (N) body_symptom
  └──→ (N) health_goal
```

### 5.4 关键概念

**diet_record + diet_record_detail 主从表**

```
diet_record (主表)              diet_record_detail (明细表)
┌──────────────────────┐       ┌──────────────────────────┐
│ id: 1                │       │ id: 1, record_id: 1      │
│ user_id: 1           │  1:N  │ food_id: 5, amount: 200  │
│ member_id: NULL      │──────→│ calories: 232            │
│ record_date: 2024-01 │       ├──────────────────────────┤
│ meal_type: lunch     │       │ id: 2, record_id: 1      │
│ total_calories: 415  │       │ food_id: 12, amount: 150 │
└──────────────────────┘       │ calories: 183            │
                               └──────────────────────────┘
```

这种设计的好处：
- 一条饮食记录可以包含多种食物
- 删除主记录时自动级联删除明细（ON DELETE CASCADE）
- 总热量在主表冗余存储，避免每次查询都 SUM 明细

**联合索引优化查询**

```sql
-- diet_record 表的索引
CREATE INDEX idx_user_date ON diet_record(user_id, record_date);
CREATE INDEX idx_user_member_date ON diet_record(user_id, member_id, record_date);
```

最常用的查询是"某用户某天的饮食记录"，联合索引让这个查询走索引扫描而不是全表扫描。

**MyBatis-Plus LambdaQueryWrapper**

```java
// 类型安全的查询条件 — 编译期就能发现字段名错误
LambdaQueryWrapper<DietRecord> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(DietRecord::getUserId, userId)        // WHERE user_id = ?
       .eq(DietRecord::getRecordDate, date)      // AND record_date = ?
       .orderByDesc(DietRecord::getCreateTime);  // ORDER BY create_time DESC
```

对比原生 MyBatis 的 XML 写法，LambdaQueryWrapper 的优势是**编译期类型检查**——如果字段名写错了，编译器会报错，而不是运行时才发现。

### 5.5 常见陷阱

| # | 陷阱 | 后果 | 避免/解决方法 |
|---|------|------|--------------|
| 1 | 忘记建索引 | 按日期查询全表扫描 | user_id + record_date 联合索引 |
| 2 | 删除主记录不级联明细 | 孤儿记录 | ON DELETE CASCADE |
| 3 | 用 SELECT * | 返回不需要的字段，浪费带宽 | 明确指定列名 |

## 六、部署架构 — Docker Compose 多容器编排

### 6.1 技术选型

| 属性 | 内容 |
|------|------|
| **技术** | Docker Compose 3.8 |
| **分类** | 容器化部署 |
| **角色** | 一键部署 MySQL + 后端 + 前端三个服务 |

### 6.2 为什么选择

**项目约束：**
- 毕业设计答辩需要演示，必须能一键启动
- MySQL、Spring Boot、Nginx 三个服务有依赖关系
- 需要保证"在我机器上能跑"在任何机器上都能跑

**权衡取舍：**
```
✅ 选择了：Docker Compose 本地编排
❌ 放弃了：Kubernetes 集群编排
⚖️ 单机部署不需要 K8s 的复杂度，Compose 足够
```

### 6.3 核心原理 — 启动流程

```
┌─────────────────────────────────────────────────┐
│  docker-compose up -d                           │
│                                                 │
│  1. 启动 mysql (healthcheck: mysqladmin ping)   │
│  2. healthcheck 通过 → 启动 backend             │
│  3. backend 启动 → 启动 frontend (Nginx)        │
│                                                 │
│  Nginx 配置:                                    │
│    /         → 前端静态文件 (Vue dist)           │
│    /api/     → 反向代理到 backend:8082           │
└─────────────────────────────────────────────────┘
```

### 6.4 关键概念

**healthcheck 保证启动顺序**

```yaml
mysql:
  healthcheck:
    test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
    interval: 10s    # 每 10 秒检查一次
    timeout: 5s      # 超时 5 秒
    retries: 5       # 重试 5 次

backend:
  depends_on:
    mysql:
      condition: service_healthy  # MySQL healthy 后才启动 backend
```

没有 healthcheck 的话，backend 可能在 MySQL 还没准备好时就尝试连接，导致启动失败。

**Nginx 反向代理**

```nginx
# 前端请求 /api/food/list → 转发到 http://backend:8082/api/food/list
location /api/ {
    proxy_pass http://backend:8082;
}
# 其他请求 → 返回 Vue 的 index.html（SPA 路由）
location / {
    try_files $uri $uri/ /index.html;
}
```

Nginx 充当了"网关"角色：静态文件自己处理，API 请求转发给后端。

## 七、核心技术决策总结

```
本项目做了 6 个关键技术决策：

1. 前后端分离 (Vue 3 + Spring Boot + REST API)
   → 解决：独立开发、交互体验
   → 代价：需要处理跨域、认证

2. JWT httpOnly Cookie 认证
   → 解决：无状态认证 + 防 XSS
   → 代价：无法主动失效 Token

3. member_id 字段数据隔离
   → 解决：一账号多成员
   → 代价：依赖代码逻辑保证隔离

4. 六维度加权评分推荐
   → 解决：可解释的个性化推荐
   → 代价：权重需手动调优

5. MyBatis-Plus ORM
   → 解决：快速 CRUD + 类型安全
   → 代价：复杂查询仍需写 SQL

6. Docker Compose 部署
   → 解决：一键部署、环境一致
   → 代价：单机部署，不支持水平扩展
```

每个决策都是在**项目约束**下做的**最优权衡**。答辩时如果评委问"为什么不用 X"，可以用 Trade-off 框架回答："X 的优势是...，但在本项目中...约束下，Y 更合适，因为..."。
