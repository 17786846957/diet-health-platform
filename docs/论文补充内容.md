## 第 4 章 系统详细设计（补充）

### 4.6 核心算法设计

#### 4.6.1 六维加权评分食物推荐算法

本系统的智能推荐模块采用六维加权评分算法，根据用户当日营养缺口、饮食历史偏好、收藏记录和当前餐次，从食物库中筛选并排序推荐 Top 10 食物。

**算法输入**：
- 用户当日已摄入营养素（热量、蛋白质、脂肪、碳水化合物）
- 用户 TDEE 目标值（由 Harris-Benedict 公式计算）
- 近 30 天食物消费频次统计
- 用户收藏食物集合
- 当前餐次类型（早餐/午餐/晚餐/加餐）

**算法流程**：

**Step 1：计算营养缺口**

系统根据用户当日已摄入的营养素与目标值的差值，按剩余餐次均分，得到每餐的营养需求：

```
need[i] = max(0, target[i] - actual[i]) / remainingMeals
```

其中 `i` 代表热量、蛋白质、脂肪、碳水化合物四种营养素。

**Step 2：SQL 预筛选**

为避免对食物库进行全表评分（当前 230+ 种食物），系统根据最大营养缺口类型进行 SQL 预筛选。若最大缺口为蛋白质，则筛选蛋白质含量 ≥ 10g 的食物；若为碳水化合物，则筛选碳水含量 ≥ 15g 的食物。预筛选结果不足 10 条时回退到全量查询。

**Step 3：六维评分**

对每种候选食物计算六个维度的得分：

| 维度 | 权重 | 计算方法 |
|------|------|----------|
| 营养适配度 | 30% | 食物 100g 营养素与缺口的匹配程度，各营养素按缺口占比加权 |
| 餐次适配度 | 25% | 食物分类与当前餐次的匹配程度（如早餐偏好主食、蛋奶） |
| 健康度评分 | 20% | 基于食物营养密度的综合评分 |
| 历史偏好 | 15% | 用户近 30 天食用频次，归一化到 [0, 1] |
| 收藏偏好 | 5% | 已收藏食物得 1.0 分，未收藏得 0 分 |
| 分类多样性 | 5% | 今日未吃过的分类得 1.0 分，已吃过则按分类数量反比计分 |

**Step 4：加权排序**

```
totalScore = nutritionScore × 0.30 + mealFitScore × 0.25
           + healthScore × 0.20 + historyScore × 0.15
           + favoriteScore × 0.05 + diversityScore × 0.05
```

按 `totalScore` 降序排列，取前 10 条返回，并附带推荐理由文本。

**关键代码**：`DietRecordService.java` 第 419-535 行的 `recommendFoods()` 方法。

#### 4.6.2 营养目标计算模型（Harris-Benedict 公式）

系统采用 Harris-Benedict 公式计算用户的基础代谢率（BMR），结合活动水平系数和健康目标调整值，得到每日目标热量。

**BMR 计算公式**：

男性：`BMR = 88.362 + 13.397 × W + 4.799 × H - 5.677 × A`

女性：`BMR = 447.593 + 9.247 × W + 3.098 × H - 4.330 × A`

其中 W 为体重（kg），H 为身高（cm），A 为年龄（岁）。

**活动水平系数**：

| 活动水平 | 系数 | 说明 |
|---------|------|------|
| sedentary | 1.200 | 久坐不动 |
| light | 1.375 | 轻度活动（每周 1-3 次） |
| moderate | 1.550 | 中度活动（每周 3-5 次） |
| active | 1.725 | 高度活动（每周 6-7 次） |
| very_active | 1.900 | 极高活动（体力劳动/专业运动） |

**目标调整**：减脂目标 -500 kcal/天，维持体重 0，增肌 +300 kcal/天。

**目标热量**：`TDEE = BMR × activityMultiplier + goalAdjustment`

**营养素分配比例**：蛋白质 15%（÷4 kcal/g）、脂肪 25%（÷9 kcal/g）、碳水化合物 60%（÷4 kcal/g）。

**关键代码**：`UserService.java` 第 164-180 行的 `calculateTargetCalories()` 方法，前端对应实现为 `utils/nutrition.js` 的 `calculateTargetCalories()` 函数。

### 4.7 安全架构设计

#### 4.7.1 JWT 认证流程

本系统采用无状态 JWT 认证机制，Token 存储于 httpOnly Cookie 中，防止 XSS 攻击窃取凭证。

**认证流程**：

1. 用户提交用户名和密码到 `POST /api/auth/login`
2. 服务端验证密码（BCrypt 哈希比对），生成 JWT（包含 userId、username、role），有效期 24 小时
3. JWT 通过 `Set-Cookie` 写入 httpOnly Cookie（名称：`diet_token`，SameSite=Lax，JavaScript 不可读）
4. 后续请求由浏览器自动携带 Cookie，`JwtAuthenticationFilter` 解析 Token 并设置 `SecurityContext`
5. Token 剩余时间不足 2 小时时自动续期，通过响应头返回新 Cookie

**Token 解析与续期关键代码**（`JwtAuthenticationFilter.java`）：

```java
Claims claims = jwtUtil.parseClaims(token);
Long userId = Long.parseLong(claims.getSubject());
String role = claims.get("role", String.class);

// Token 即将过期时自动续期
long remaining = claims.getExpiration().getTime() - System.currentTimeMillis();
if (remaining < RENEW_THRESHOLD_MS) {
    String newToken = jwtUtil.generateToken(userId, username, role);
    addTokenCookie(request, response, newToken);
}
```

**Cookie 设置**（`SameSite=Lax` 防止 CSRF，`HttpOnly` 防止 XSS）：

```java
response.addHeader("Set-Cookie",
    "diet_token=" + token + "; Path=/; HttpOnly; Max-Age=86400; SameSite=Lax" + secure);
```

**前端 Token 管理**：前端不存储 Token，仅缓存用户基本信息用于 UI 显示。`request.js` 配置 `withCredentials: true` 使 Axios 自动携带 Cookie。401 响应时自动清除本地状态并跳转登录页。

#### 4.7.2 接口限流机制

系统对登录和注册接口实施基于 IP 的频率限制，防止暴力破解和恶意注册。

| 接口 | 限制 | 时间窗口 |
|------|------|----------|
| POST /auth/login | 5 次/分钟 | 滑动窗口 |
| POST /auth/register | 3 次/2分钟 | 滑动窗口 |

实现采用 `ConcurrentHashMap` 存储每个 IP 的请求计数，通过 `@Scheduled` 定时任务每 5 分钟清理过期记录，防止内存泄漏。

**关键代码**（`RateLimitFilter.java`）：

```java
private static final int MAX_LOGIN_ATTEMPTS = 5;
private static final long WINDOW_MS = 60 * 1000;

AttemptInfo info = loginAttemptMap.compute(clientIp, (key, existing) -> {
    if (existing == null || System.currentTimeMillis() - existing.windowStart > WINDOW_MS) {
        return new AttemptInfo();
    }
    return existing;
});
if (info.attempts.incrementAndGet() > MAX_LOGIN_ATTEMPTS) {
    response.setStatus(429);
    response.getWriter().write(R.error(429, "登录尝试过于频繁，请1分钟后再试"));
    return false;
}
```

### 4.8 家庭成员数据隔离设计

#### 4.8.1 数据库层隔离

系统通过 `member_id` 字段实现同一用户账号下多成员的数据隔离。涉及的表包括：`diet_record`、`water_record`、`exercise_record`、`weight_record`、`health_goal`、`body_symptom`。

查询时通过 `userMemberWrapper` 方法构建查询条件：

```java
private LambdaQueryWrapper<DietRecord> userMemberWrapper(Long userId, Long memberId) {
    LambdaQueryWrapper<DietRecord> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(DietRecord::getUserId, userId);
    if (memberId == null) {
        wrapper.isNull(DietRecord::getMemberId);  // 用户本人数据
    } else {
        wrapper.eq(DietRecord::getMemberId, memberId);  // 家庭成员数据
    }
    return wrapper;
}
```

当 `memberId` 为 null 时查询用户本人数据（`member_id IS NULL`），非 null 时查询指定家庭成员数据。此模式在 6 个服务中统一使用，确保数据隔离的一致性。

#### 4.8.2 前端状态管理

前端通过 Pinia 的 `useUserStore` 管理当前活跃的成员上下文：

```javascript
const activeMember = ref(null)       // 当前选中的家庭成员
const activeMemberId = computed(() => activeMember.value?.id || null)
const activeProfile = computed(() => activeMember.value || user.value)

function setActiveMember(member) {
  activeMember.value = member
  if (member) {
    localStorage.setItem('activeMember', JSON.stringify(member))
  } else {
    localStorage.removeItem('activeMember')
  }
}
```

`activeProfile` 是一个计算属性，优先返回选中的家庭成员信息，未选中时返回用户自身信息。所有营养目标计算、饮食记录查询等操作均基于 `activeProfile` 获取当前操作对象。

### 5.1 饮水记录模块实现

饮水记录模块提供每日饮水量的记录、查询和统计功能，支持按日期筛选和趋势可视化。

**数据模型**：`WaterRecord` 实体包含用户 ID、成员 ID、记录日期、饮水量（ml）和创建时间。

**核心功能**：
- 添加记录：校验日期和饮水量（1-5000ml），通过 `userMemberWrapper` 确保数据归属正确
- 按日查询：返回当日所有饮水记录列表
- 统计汇总：计算当日总饮水量和记录次数
- 趋势图表：ECharts 折线图展示近 7 天/30 天饮水趋势

**API 设计**：

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | /api/water | 添加饮水记录 |
| DELETE | /api/water/{id} | 删除记录 |

### 5.2 运动记录模块实现

运动记录模块支持多种运动类型的记录，内置基于 MET（代谢当量）的热量消耗估算。

**MET 热量计算**：系统根据运动类型匹配 MET 值，结合用户体重和运动时长计算消耗热量：

```
热量消耗(kcal) = MET × 体重(kg) × 时长(小时)
```

运动类型匹配采用中英文双语识别，支持跑步、步行、游泳、骑行、瑜伽、力量训练等常见运动。

**API 设计**：

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | /api/exercise | 添加运动记录 |
| PUT | /api/exercise/{id} | 更新记录 |

| GET | /api/exercise/list?date=YYYY-MM-DD | 按日查询 |
| GET | /api/exercise/stats?date=YYYY-MM-DD | 当日统计 |

### 5.3 体重记录模块实现

体重记录模块支持体重和体脂率的跟踪记录，提供趋势图表和数据对比功能。

**特殊设计**：体重记录采用 Upsert 模式——同一日期的记录自动合并而非创建新记录，避免用户重复记录导致数据冗余。

### 5.6 前端状态管理架构

#### 5.6.1 Pinia Store 设计

系统采用 Pinia 进行状态管理，按功能领域划分为 3 个 Store：

| Store | 职责 | 核心状态 |
|-------|------|----------|
| `useUserStore` | 认证、用户信息、家庭成员管理 | `user`、`activeMember`、`familyMembers` |
| `useAppStore` | 全局 UI 状态 | `loading`（并发请求计数器） |
| `useDietStore` | 饮食记录相关状态 | `records`、`dailyStats`、`weeklyStats` |

**用户 Store 关键设计**：

```javascript
export const useUserStore = defineStore('user', () => {
  const user = ref(null)                    // 用户基本信息
  const activeMember = ref(null)            // 当前选中的家庭成员
  const isLoggedIn = computed(() => !!user.value)
  const activeProfile = computed(() => activeMember.value || user.value)

  async function login(form) {
    const res = await loginApi(form)
    user.value = res.data.user              // Token 由 httpOnly Cookie 管理
    localStorage.setItem('user', JSON.stringify(res.data.user))
  }

  async function logout() {
    try { await logoutApi() } catch {}
    finally {
      user.value = null
      activeMember.value = null
      localStorage.removeItem('user')
      localStorage.removeItem('activeMember')
    }
  }
})
```

### 5.7 API 请求层实现

前端通过 Axios 封装统一的 API 请求层，实现以下功能：

**请求拦截器**：通过并发请求计数器管理全局 loading 状态，避免多个请求并发时 loading 闪烁。

**响应拦截器**：
1. 自动解包 `R<T>` 统一响应格式（`{code, message, data}`）
2. `code !== 200` 时 reject Promise，由调用方处理错误
3. 401 响应时自动清除本地状态、显示错误提示、1.5 秒后跳转登录页

**关键代码**（`src/api/request.js`）：

```javascript
const request = axios.create({
  baseURL: '/api',
  timeout: 10000,
  withCredentials: true  // 自动携带 httpOnly Cookie
})

let activeRequests = 0

request.interceptors.response.use(
  response => {
    const { data } = response
    if (data.code !== 200) {
      return Promise.reject(new Error(data.message || '请求失败'))
    }
    return data
  },
  async error => {
    if (error.response?.status === 401) {
      const userStore = useUserStore()
      userStore.logout()
      ElMessage.error('未登录或登录已过期')
      setTimeout(() => { window.location.href = '/login' }, 1500)
    }
    return Promise.reject(new Error(msg))
  }
)
```

### 5.8 全局异常处理实现

后端通过 `@RestControllerAdvice` 实现统一异常处理，将业务异常映射为标准 HTTP 响应：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public R<?> handleBusinessException(BusinessException e, HttpServletResponse response) {
        HttpStatus status = HttpStatus.BAD_REQUEST;
        switch (e.getCode()) {
            case 401: status = HttpStatus.UNAUTHORIZED; break;
            case 403: status = HttpStatus.FORBIDDEN; break;
            case 404: case 1002: case 1004: case 1005: status = HttpStatus.NOT_FOUND; break;
        }
        response.setStatus(status.value());
        return R.error(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public R<?> handleValidation(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));
        return R.error(400, message);
    }
}
```

**统一响应格式** `R<T>`：

```java
public class R<T> {
    private int code;       // 200 成功，其他为错误码
    private String message; // 响应消息
    private T data;         // 响应数据
}
```
