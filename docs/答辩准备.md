## 二、开场陈述讲稿

### Slide 1：封面（30 秒）

> 各位老师好！我是计算机科学与技术专业的XXX。毕业设计题目是"基于 Spring Boot 的智能饮食健康管理平台的设计与实现"，指导老师是XXX。下面从选题背景、技术方案、系统设计、创新点、测试验证五个方面汇报。

### Slide 2：选题背景（1 分钟）

> 随着"健康中国 2030"战略推进，科学化饮食管理成为大众需求。现有应用如 MyFitnessPal、薄荷健康存在三个局限：一是大多面向单用户设计，不支持家庭成员独立管理；二是功能维度单一，只关注饮食而忽略运动、饮水等健康管理；三是推荐算法简单，缺乏对用户历史偏好和营养缺口的综合考量。因此本系统有两个核心目标：构建家庭多成员数据隔离的全维度健康管理平台；设计多维度加权推荐算法。

### Slide 3：技术架构（1 分钟）

> 系统采用前后端分离 B/S 架构，分为三层。表现层：Vue 3 + Vite + Element Plus + ECharts，负责页面渲染和数据可视化。业务层：Spring Boot 2.7 + MyBatis-Plus + Spring Security，负责业务逻辑和安全认证。数据层：MySQL 8.0，共 11 张业务表。部署用 Docker Compose 编排三个容器，一条命令完成部署。

### Slide 4：系统设计（1 分钟）

> 系统共 13 个功能模块、50 余个 RESTful API、11 张数据库表。核心设计有两个：第一，diet_record 表的 member_id 字段实现家庭成员数据隔离，避免拆表导致查询复杂；第二，接口统一 R 响应格式（code、message、data），全局异常处理器覆盖各类异常。

### Slide 5：核心功能（1.5 分钟）

> 三个核心功能。第一，JWT 认证：Token 存 httpOnly Cookie，JavaScript 读不到防 XSS，SameSite=Lax 防 CSRF，剩余 2 小时自动续期。第二，智能推荐：六维度加权评分算法，营养适配度 30% + 餐次适配度 25% + 食物健康度 20% + 历史偏好 15% + 收藏 5% + 多样性 5%，综合评估食物与用户的匹配程度。第三，营养目标：Harris-Benedict 公式计算基础代谢率，乘活动系数和目标调整值得到每日热量目标。

### Slide 6：创新点（1 分钟）

> 四个创新点。第一，家庭成员数据隔离机制，一个账号管理多人独立饮食数据。第二，六维度加权推荐算法，不仅考虑营养缺口，还综合餐次适配、食物健康度、历史偏好、收藏和多样性。第三，JWT httpOnly Cookie + SameSite 安全架构，兼顾防 XSS 和防 CSRF。第四，全维度健康管理，整合饮食、运动、饮水、体重、症状五大维度。

### Slide 7：测试（30 秒）

> 测试分三层。单元测试 21 个，Mock 依赖验证业务逻辑；集成测试 30 个，MockMvc 模拟 HTTP 请求；E2E 测试 34 个，Playwright 模拟真实用户操作。共 85 个用例，全部通过，通过率 100%。

### Slide 8：总结（30 秒）

> 以上就是汇报。主要成果：前后端分离的饮食健康管理平台，13 个功能模块，50 余个 API 接口，85 个测试用例。核心技术方案：家庭成员数据隔离和六维度加权推荐算法。恳请各位老师批评指正，谢谢！

## 四、常见问题及回答

### Q1: 和市面上的饮食 App 有什么区别？

> 四点区别。第一，支持家庭成员管理，一个账号管理多人独立饮食数据。第二，推荐算法综合六个维度加权评分，不是简单热量计算。第三，全维度健康管理，整合饮食、运动、饮水、体重、症状五大维度。第四，JWT httpOnly Cookie 安全方案，防 XSS 和 CSRF。

### Q2: 家庭成员数据隔离怎么实现？

> 数据库层：diet_record 表加 member_id 字段，NULL 表示主用户，有值表示家庭成员。所有查询用统一的 LambdaQueryWrapper 条件构建方法。前端层：Pinia 的 activeMember 状态管理当前成员，router-view 动态 key 强制重新挂载页面。这种方案比拆表更简洁，查询效率也更高。

### Q3: 推荐算法怎么设计的？

> 六维度加权评分策略。营养适配度 30%——评估食物能否填补当前蛋白质/脂肪/碳水缺口。餐次适配度 25%——早餐适合主食/蛋奶，午晚餐适合肉类/蔬菜。食物健康度 20%——高蛋白低热量加分，纯糖/高脂惩罚。历史偏好 15%——统计近 30 天消费频次。收藏偏好 5%——收藏过的食物加分。分类多样性 5%——鼓励尝试不同类别。加权求和排序返回 Top 10。

### Q4: 热量目标怎么计算？

> Harris-Benedict 公式算基础代谢率 BMR，男性：88.362 + 13.397×体重 + 4.799×身高 - 5.677×年龄，女性：447.593 + 9.247×体重 + 3.098×身高 - 4.330×年龄。乘活动系数（久坐 1.2 到极活跃 1.9），加目标调整值（减脂 -500，维持 0，增肌 +300）。营养素按蛋白质 15%、脂肪 25%、碳水 60% 分配。

### Q5: 为什么用 JWT 不用 Session？

> 三个原因。第一，JWT 无状态，不需要服务端存会话，适合前后端分离和未来分布式部署。第二，Token 存 httpOnly Cookie，JavaScript 读不到，防 XSS。第三，SameSite=Lax 属性防 CSRF。传统 Session 需要服务端存储，在分布式场景下需要额外的 Session 共享方案。

### Q6: 系统有哪些安全措施？

> 九个方面。BCrypt 密码加密；JWT httpOnly Cookie 存 Token；登录限流 5 次/分钟防暴力破解；注册限流 3 次/2 分钟防刷；管理员角色权限控制；记录归属校验（只能操作自己的数据）；@Valid 参数校验；全局异常处理不泄露内部堆栈信息；审计日志通过 @AuditLog 注解 + AOP 切面记录关键操作。

### Q7: 数据库设计有哪些考虑？

> 11 张表。核心表 6 张（用户、食物、饮食记录、记录明细、家庭成员、食物收藏），扩展表 5 张（健康目标、饮水记录、运动记录、体重记录、身体症状）。关键设计：diet_record 的 member_id 字段实现数据隔离，避免拆表；diet_record_detail 批量插入优化写入性能；user_id + record_date 联合索引加速按日期查询。

### Q8: 项目遇到什么困难？

> 三个典型问题。第一，Cookie 的 SameSite 属性 Servlet API 不直接支持，改用 Set-Cookie 响应头手动拼接解决。第二，家庭成员数据隔离最初想单独建表，后来发现用 member_id 字段更简洁，查询也更高效。第三，推荐算法从最初的简单热量匹配迭代到六维度加权评分，每增加一个维度都需要调整权重平衡。

### Q9: 如果继续完善会做什么？

> 三个方向。第一，推荐算法引入协同过滤，利用其他用户的行为数据提升推荐质量。第二，开发移动端 App，用 uni-app 跨平台方案。第三，引入 Redis 缓存热点数据（食物库、用户统计），减少数据库查询。

### Q10: 什么是前后端分离？有什么优势？

> 前端 Vue 和后端 Spring Boot 独立开发部署，只通过 RESTful API 通信。三大优势：开发效率高，前后端并行开发互不阻塞；部署灵活，前端静态文件可放 CDN；可扩展性强，同一后端可服务 Web、小程序、App 多种客户端。

### Q11: MyBatis-Plus 相比 MyBatis 有什么优势？

> 三个核心优势。第一，通用 CRUD 方法自动生成，继承 BaseMapper 就有增删改查，减少重复代码。第二，LambdaQueryWrapper 类型安全的条件构造器，编译期就能发现字段名错误。第三，内置分页插件，配合 Page 对象实现物理分页。

### Q12: 运动热量消耗怎么计算的？

> 基于 MET（代谢当量）值计算。MET 值表示运动时能量消耗相对于静息代谢的倍数，比如慢跑约 7 MET，游泳约 8 MET。热量消耗 = MET 值 × 体重(kg) × 时间(小时)。系统根据运动类型和强度查找对应的 MET 值，自动计算消耗热量。

### Q13: Docker Compose 的作用是什么？

> 用 YAML 文件定义和运行多容器应用。本系统编排三个服务：MySQL、Spring Boot 后端、Nginx 前端。通过 depends_on + healthcheck 确保启动顺序（MySQL 就绪后才启动后端）。一条 docker-compose up -d 命令完成所有服务的部署，环境一致性有保障。

### Q14: 前后端 BMR 公式一致吗？

> 一致，前后端都使用 Harris-Benedict 修正公式。前端在 utils/nutrition.js 的 calculateBMR 函数中实现，用于页面展示和热量预警。后端在 UserService 的 buildTargets 方法中实现，用于营养缺口计算和推荐算法。两套代码使用相同的系数，保证前后端数据一致。

### Q15: 系统的前后端怎么通信的？

> 前端用 axios 封装 HTTP 请求，统一放在 src/api/ 目录。请求携带 withCredentials: true 自动发送 httpOnly Cookie。响应拦截器处理 401（跳转登录页）和统一错误提示。后端用 @RestController + @RequestMapping 定义 RESTful 接口，返回统一的 R 对象（code、message、data）。

## 六、核心知识点速查

| 知识点 | 关键词 | 一句话解释 |
|--------|--------|-----------|
| JWT | 无状态、httpOnly Cookie | Token 存 Cookie，JS 读不到防 XSS |
| BCrypt | 单向哈希、加盐 | 密码加密存储，不可逆 |

| Docker Compose | YAML、depends_on | 多容器编排，一条命令部署 |
| Harris-Benedict | BMR、活动系数 | 基础代谢率公式，计算每日热量目标 |
| MET 值 | 代谢当量 | 运动热量消耗 = MET × 体重 × 时间 |
| 前后端分离 | B/S、REST API | 前端 Vue + 后端 Spring Boot，API 通信 |
| member_id | 数据隔离 | NULL=主用户，有值=家庭成员，查询自动过滤 |
