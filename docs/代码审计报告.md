## 二、Critical 问题（必须立即修复）

### BUG-001: DietAdviceView.vue 缺少 ElMessage 导入

- **位置**: `diet-health-frontend/src/views/DietAdviceView.vue` 第 333、350、366 行
- **问题**: 调用了 `ElMessage.error()` 但未在 `<script setup>` 中导入 `ElMessage`。Element Plus 全局安装不会自动注册服务函数到 `<script setup>` 作用域。
- **影响**: 当每日建议、饮食分析、健康建议三个 API 调用任一失败时，触发 `ReferenceError`，用户看到空白页面而非错误提示。
- **修复**: 在 `<script setup>` 顶部添加 `import { ElMessage } from 'element-plus'`

### BUG-002: WaterRecordView.vue UTC 日期 Bug

- **位置**: `diet-health-frontend/src/views/WaterRecordView.vue` 第 118 行
- **问题**: `new Date().toISOString().split('T')[0]` 返回 UTC 时间。中国时区 UTC+8 下，16:00 后 UTC 日期比本地日期晚一天。
- **影响**: 下午 4 点后打开饮水记录页面，显示的是"昨天"的记录，用户误以为当天没有数据。
- **修复**: 改用项目已有的 `getToday()` 工具函数（来自 `src/utils/date.js`），该函数使用本地时间。

### BUG-003: ExerciseRecordView.vue UTC 日期 Bug

- **位置**: `diet-health-frontend/src/views/ExerciseRecordView.vue` 第 120 行
- **问题**: 与 BUG-002 完全相同。
- **修复**: 同 BUG-002。

### BUG-004: Docker Compose 缺少 expand_tables.sql

- **位置**: `docker-compose.yml` 第 16-17 行
- **问题**: volumes 挂载了 `init.sql` 和 `food_expand.sql`，但遗漏了 `expand_tables.sql`。该文件包含 `health_goal`、`water_record`、`exercise_record`、`body_symptom`、`weight_record` 五张表的建表语句。
- **影响**: `docker-compose up -d` 后，饮水、运动、体重、健康目标、身体症状五个功能模块全部不可用。
- **修复**: 添加 `- ./sql/expand_tables.sql:/docker-entrypoint-initdb.d/03-expand-tables.sql`

### BUG-005: diet_record 外键缺少 ON DELETE CASCADE

- **位置**: `sql/init.sql` 第 49 行
- **问题**: `fk_diet_user` 约束没有 `ON DELETE CASCADE`。删除用户时，数据库抛出外键约束异常，无法删除。
- **影响**: 管理员删除用户功能在数据库层面会失败。
- **修复**: 修改为 `CONSTRAINT fk_diet_user FOREIGN KEY (user_id) REFERENCES user(id) ON DELETE CASCADE`

## 三、High 问题（架构 / 安全 / 维护性）

### SEC-001: CSRF 禁用但使用 Cookie 认证

- **位置**: `diet-health-backend/src/main/java/com/diet/config/SecurityConfig.java` 第 37 行
- **问题**: `.csrf().disable()` 在纯 JWT Header 场景下是安全的，但本项目使用 httpOnly Cookie 存储 Token，浏览器会自动附带 Cookie，攻击者可构造恶意页面发起请求。
- **缓解**: `SameSite=Lax` 提供了部分保护（阻止跨站 POST），但不防御 GET 型状态变更。
- **建议**: 对写操作（POST/PUT/DELETE）增加 CSRF Token 校验，或改用 `SameSite=Strict`。

### SEC-002: X-Real-IP 可伪造绕过限流

- **位置**: `diet-health-backend/src/main/java/com/diet/config/RateLimitFilter.java` 第 114-121 行
- **问题**: `getClientIp()` 读取 `X-Real-IP` 请求头。没有反向代理覆盖时，客户端可伪造该头绕过 IP 限流。
- **修复**: 优先读取 `request.getRemoteAddr()`，仅在可信代理环境下才信任 `X-Real-IP`。

### SEC-003: 生产配置 useSSL=false

- **位置**: `application-prod.yml` 第 8 行
- **问题**: 数据库连接禁用了 SSL，生产环境存在中间人攻击风险。
- **修复**: 生产环境启用 `useSSL=true` 并配置 CA 证书。

### ARCH-001: DTO-Entity 映射在每个控制器重复

- **位置**: 所有 13 个 Controller 的 add/update 方法
- **问题**: 手动逐字段复制 DTO → Entity，代码大量重复。例如 `DietRecordController` 第 37-51 行和第 60-76 行的映射逻辑完全相同。
- **影响**: 新增字段时必须修改多处，遗漏会导致数据丢失。
- **建议**: 使用 MapStruct 或在 DTO 中添加 `toEntity()` 方法。

### ARCH-002: buildWrapper() 在 6 个服务中复制

- **位置**: `DietRecordService`、`ExerciseRecordService`、`WaterRecordService`、`WeightRecordService`、`HealthGoalService`、`BodySymptomService`
- **问题**: 根据 userId + memberId 构建 LambdaQueryWrapper 的逻辑在 6 个服务中完全相同。
- **修复**: 提取到共享工具类或基础服务类。

### ARCH-003: recommendFoods() 方法 217 行

- **位置**: `DietRecordService.java` 第 419-758 行
- **问题**: 包含营养缺口计算、历史偏好查询、SQL 预筛选、6 维评分、排序输出等 7 个逻辑步骤，违反单一职责原则。
- **建议**: 提取为独立的 `FoodRecommendationService`。

### ARCH-004: DataConsistencyService 缺少 @Transactional

- **位置**: `DataConsistencyService.java` 第 46 行
- **问题**: `verifyTotalCalories()` 方法读写数据（更新 totalCalories）但没有事务注解。中途失败会导致部分记录更新、部分未更新。
- **修复**: 添加 `@Transactional(rollbackFor = Exception.class)`

### TEST-001: 9/13 控制器无集成测试

- **位置**: `diet-health-backend/src/test/java/com/diet/controller/`
- **问题**: 仅有 `AuthControllerTest`、`DietRecordControllerTest`、`FamilyMemberControllerTest`、`FoodControllerTest`。缺少：WaterRecord、ExerciseRecord、WeightRecord、HealthGoal、BodySymptom、Favorite、User、DietAdvice、Admin 共 9 个控制器的测试。
- **影响**: 答辩时如果老师问测试覆盖率，这些模块无法展示。

### TEST-002: FamilyMemberService 无单元测试

- **位置**: `diet-health-backend/src/main/java/com/diet/service/FamilyMemberService.java`
- **问题**: 核心业务逻辑（10 人上限、所有权校验）没有独立的单元测试。`FamilyMemberControllerTest` 通过 MockMvc 间接测试，但无法覆盖服务层边界情况。

### DB-001: 缺少 CHECK 约束

- **位置**: `sql/init.sql`、`sql/expand_tables.sql`
- **问题**: `gender`、`activity_level`、`goal`、`severity` 等枚举型字段没有数据库层约束。`severity` 注释写"1-5"但实际允许任意整数。
- **修复**: 为每个枚举字段添加 CHECK 约束。

### DB-002: 缺少关键索引

- **位置**: `sql/init.sql`
- **问题**:
  - `food.category` — 食物库分类筛选是高频操作，无索引导致全表扫描
  - `user.role` — 管理员仪表盘按角色查询，无索引
  - `diet_record_detail.food_id` — 推荐算法按食物 ID 查询历史频次，无索引
- **修复**: 添加对应索引。
